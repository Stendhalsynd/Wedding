# S1-5 배포 마무리 및 프로젝트 문서화

## 목표
- 실제 예비신랑/예비신부 PNG 자산이 빌드와 배포 결과에 반영되어야 한다.
- README는 현재 프로젝트의 목적, 주요 기능, 로컬 실행, 배포 전제 조건을 설명해야 한다.
- GitHub Release 업로드는 현재 저장소가 제공하는 산출물 범위에 맞춰 수행 가능 여부를 명확히 해야 한다.

## 전제
- `public/예비신랑.png`, `public/예비신부.png`는 사용자가 제공한 실제 자산 파일이어야 한다.
- 현재 저장소는 웹 앱(Vite + React)이며 Android 빌드 구조는 없다.

## 동작 규칙
- 프로필 기본 이미지는 항상 `/예비신랑.png`, `/예비신부.png`를 사용한다.
- 배포 후 정적 파일이 새 자산으로 서빙되는지 확인한다.
- README에는 다음을 포함한다.
  - 프로젝트 소개
  - 핵심 기능
  - 개발 환경 실행 방법
  - Firebase Authorized Domains 및 Vercel 배포 메모
- GitHub Release는 현재 범위에서 다음 둘 중 하나만 허용한다.
  - 이미 존재하는 release APK를 업로드
  - APK가 없으면 웹 배포 산출물만 완료로 보고 APK 업로드는 보류한다

## 산출물
- README 업데이트
- `/plan` 진행 로그 업데이트
- 배포 검증 결과

## 검증 기준
- `npm run build`가 통과한다.
- Vercel production 재배포가 통과한다.
- 배포 URL에서 실제 PNG 자산이 갱신된 상태로 서빙된다.
- README가 현재 프로젝트 설명과 운영 전제 조건을 반영한다.

## 리스크
- APK는 현재 리포지토리만으로 빌드할 수 없다.
- GitHub push / release 업로드는 네트워크 권한과 계정 인증 상태가 필요하다.
