# S2-0 Android Packaging Program Execution Model

## 목표
- S2 Android 패키징 작업은 역할 기반 subagent 분산 실행을 기본 원칙으로 삼는다.
- S2 각 슬라이스는 스펙, 구현, 검증, 리뷰가 분리된 상태로 진행되며, 병렬 가능한 작업만 병렬로 처리한다.
- 이후 구현 세션에서도 동일한 실행 규칙을 재사용할 수 있어야 한다.

## 기준
- SDD는 필수다. 각 슬라이스는 구현 전에 `/spec/S2-x.md`를 먼저 작성한다.
- 진행 상태는 `/plan/*.md`에 기록한다.
- 역할 분리는 [VoltAgent/awesome-codex-subagents](https://github.com/VoltAgent/awesome-codex-subagents)의 책임 분리 방식을 참고한다.

## 역할 정의
- `dependency-expert`
  - 공식 문서, SDK, 프레임워크, 플랫폼 제약 검토
- `architect`
  - 경계, 인터페이스, 마이그레이션, 런타임 책임 분리 검토
- `executor`
  - 실제 구현
- `test-engineer`
  - 테스트, acceptance criteria, verification scenario 정의
- `debugger`
  - 빌드/런타임 이슈 분석
- `qa-tester`
  - Playwright 또는 수동 검증 흐름 점검
- `writer`
  - README, release, 운영 문서 정리
- `code-reviewer`
  - 슬라이스 완료 후 review gate

## 병렬화 규칙
- 병렬화는 서로 독립된 읽기/분석/검증 작업에만 적용한다.
- 같은 파일을 동시에 수정하는 구현 작업은 금지한다.
- 한 슬라이스에는 하나의 주 구현 agent만 둔다.
- 병렬 실행 권장 대상:
  - 공식 문서 조사
  - 현재 코드 구조 분석
  - 테스트 시나리오 정리
  - QA 준비
  - 리뷰
- 순차 실행 필수 대상:
  - 기반 Android 구조 생성 전 Android 인증 구현
  - signed APK 생성 전 release 업로드

## 슬라이스별 기본 실행 순서
1. `/spec/S2-x.md` 작성
2. `/plan/*.md` 진행 로그 갱신
3. `dependency-expert` 실행
4. 필요 시 `architect` 실행
5. `executor` 구현
6. `test-engineer` 및 `qa-tester` 검증
7. `code-reviewer` review gate
8. review 이슈 반영
9. 최종 검증 후 다음 슬라이스 이동

## S2 역할 매핑

### S2-1 Capacitor Android Base
- `dependency-expert`: Capacitor Android 추가, config, asset sync 공식 문서 검토
- `architect`: 웹 앱과 Android shell 경계 정리
- `executor`: `capacitor.config.ts`, npm scripts, Android 프로젝트 생성
- `code-reviewer`: 설정/구조 리뷰

### S2-2 Android Google Login
- `dependency-expert`: Firebase Android + Capacitor Firebase Auth 공식 흐름 검토
- `architect`: `skipNativeAuth + signInWithCredential` 구조 검증
- `executor`: auth facade, login flow 변경
- `test-engineer`: web/android 분기 테스트 설계
- `code-reviewer`: 인증 상태/회귀 리뷰

### S2-3 Signing / APK Build
- `dependency-expert`: Android signing / release build 공식 문서 검토
- `executor`: Gradle signing config, release build script 구현
- `test-engineer`: release artifact guard / version rule 검증
- `code-reviewer`: secret handling / signing config 리뷰

### S2-4 App Identity Icons
- `designer` 또는 `writer`: 아이콘 방향 및 자산 규칙 정리
- `executor`: 웹 favicon / Android launcher icon 생성 및 적용
- `qa-tester`: 웹 자산 로드 및 Android 리소스 존재 확인
- `code-reviewer`: 자산 경로 / 설정 리뷰

### S2-5 Tag / GitHub Release / Discord
- `executor`: release orchestration script 구현
- `writer`: README / release usage 문서 정리
- `qa-tester`: dry-run / failure-path / happy-path 검증
- `code-reviewer`: release workflow 리뷰

## 산출물 규칙
각 subagent 결과는 다음을 포함해야 한다.
- 무엇을 확인/구현했는지
- 근거 파일 또는 공식 문서
- 남은 리스크 또는 follow-up
- 구현 agent면 수정 대상 파일 목록
- 리뷰 agent면 severity 포함 findings

## 완료 기준
- S2 각 슬라이스는 spec, plan, verification, review evidence가 모두 남아 있어야 한다.
- 구현 완료 보고 시 미결정 사항이 남아 있으면 안 된다.
- 다음 슬라이스로 넘어가기 전에 review issue가 모두 반영되어야 한다.
